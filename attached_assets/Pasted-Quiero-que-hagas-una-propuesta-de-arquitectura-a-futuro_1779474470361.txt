# Requerimientos arquitectónicos

## 1. Inventario real (prioridad máxima)

Analiza y propone la mejor transición desde:

```text
producto como texto libre
```

hacia:

```text
productos reales normalizados
```

Pero evitando romper:

* ventas actuales
* compras actuales
* historial existente

Quiero una estrategia progresiva y segura.

Evalúa:

* tabla productos
* stock automático
* kardex
* costos promedio
* relación compras ↔ ventas
* integridad referencial
* migración gradual

Y define:

* qué columnas deben permanecer temporalmente
* qué columnas deben eliminarse después
* cómo mantener compatibilidad durante transición

---

## 2. Pagos y caja

Analiza completamente el flujo financiero actual.

Detecta:

* duplicación de datos
* cálculos incorrectos
* campos redundantes
* riesgos contables

Quiero eliminar hacks como:

```text
"efectivo,yape"
```

y reemplazarlo por una arquitectura correcta.

Pero evita crear demasiadas tablas.

Analiza cuál es la mínima estructura necesaria para:

* pagos múltiples
* pagos mixtos
* caja exacta
* trazabilidad financiera
* conciliación futura

Evalúa si:

* pagos debe ser la fuente de verdad
* movimientos debe convertirse en ledger financiero central
* metodo_pago debe salir de ventas
* movimientos puede reemplazar historiales separados

---

## 3. Integridad de base de datos

Haz una auditoría estructural.

Detecta:

* relaciones débiles
* riesgos de datos huérfanos
* borrados peligrosos
* inconsistencias potenciales
* campos desnormalizados
* duplicaciones innecesarias

Evalúa:

* foreign keys faltantes
* ON DELETE correctos
* uso de soft deletes
* índices necesarios
* constraints importantes
* transacciones necesarias

Quiero evitar corrupción silenciosa de datos.

---

## 4. Edición segura de ventas

Analiza el flujo actual donde:

```php
delete + recreate
```

rompe relaciones.

Propón una solución:

* limpia
* eficiente
* mantenible
* sin lógica compleja innecesaria

---

## 5. Performance y escalabilidad

Analiza:

* N+1 queries
* relaciones dinámicas
* eager loading
* cálculos repetidos
* atributos computados costosos

Propón:

* estructura correcta de queries
* scopes
* caché solo si realmente vale la pena
* evitar lógica pesada en modelos

Quiero que el sistema pueda crecer sin volverse lento.

---

## 6. Precisión financiera

Analiza:

* uso actual de decimales
* offsets manuales 0.005
* riesgos de precisión

Propón:

* solución ideal
* solución pragmática
* costo/beneficio real

No quiero complejidad innecesaria, pero tampoco errores financieros ocultos.
